The Engineering Challenge

Naively copying files to the cloud fails instantly on Windows. SyncVault was built to bypass the two biggest hurdles in daemon-based synchronization.

🔒

The Win32 File Lock Problem

When an application writes to a file, the Windows OS places an exclusive lock on it. Standard sync scripts crash with a filesystem_error when encountering these. SyncVault implements a retry-backoff queue that gracefully polls file handles until the OS safely releases them.

🧠

The Memory Drift Problem

If a sync engine tracks file states solely in a RAM vector, closing the app or a sudden power loss wipes that state clean—causing cloud data desyncs on reboot. By offloading tracker state entirely to SQLite, SyncVault guarantees zero structural drift across restarts.

Core Architecture

Under the hood details on how reliability is maintained under state failure.

🗄️

Relational State Management

SyncVault commits system file metrics (isDir, isDeleted, path) into a transaction-safe embedded SQLite database. This tombstoning methodology isolates local delete steps to protect cloud mirrors from data drift.

🛡️

Fault Tolerance & Recovery

Designed for chaos. If a crash compromises the database file, the daemon detects the corruption signature instantly, initializes an emergency filesystem trace, and reconciles structural trees directly from cloud manifests.

⚙️

Dynamic Configuration

Zero downtime updates. Target storage reallocations adjust runtime targets without interrupting active multi-threaded synchronization queues. Updates propagate via thread-safe memory hooks, eliminating process teardowns.

Multithreaded Topology

How concurrent processes execute without blocking the user's operating system.

Thread Component Core Responsibility Coordination Mechanism
Main Daemon Thread Initializes the background service, reads JSON configurations, and monitors overall system health. Spawns and manages the lifecycle of worker sub-processes.
FS Watcher Thread Listens natively to directory change notifications via the Windows kernel (avoiding heavy CPU polling). Pushes altered file paths into a std::queue protected by mutexes.
Sync Worker Pool Picks up tasks from the queue, checks SQLite DB states, resolves file locks, and executes cloud uploads. Utilizes std::condition_variable to sleep when idle, preventing CPU spinning.

The Data Lifecycle

Tracing the path of a deleted file from local disk to the cloud.

01

Kernel Detection

The user deletes a local file. The Windows Kernel flags a filesystem event. The FS Watcher Thread catches this event and pushes the file path into the thread-safe Task Queue.

02

SQLite Tombstoning

A Sync Worker wakes up, pulls the task, and updates the local SQLite database. Instead of dropping the row, it flips IsDeleted = 1. This ensures a persistent record of the action exists even if the system crashes immediately after.

03

Cloud Reconciliation

The Sync Worker securely connects to the remote cloud API, matches the file hash or ID, and executes the remote deletion request.

04

State Pruning

Upon receiving an HTTP 200 OK from the cloud, the worker permanently removes the tombstoned row from the SQLite database, completing the synchronization cycle seamlessly.
